매번 만들지만 매번 다른 구현 방식을 고민하게 되는 Dialog를 만들어보자.
 

Dialog 컴포넌트 추상화하기

Dialog 컴포넌트를 설계할 때, 중요한 것은 재사용성과 확장성이다. 서비스의 성장과 함께 Dialog의 유형이 다양해질 수 있으므로, 컴포넌트를 어느정도 유연하게 만드는 것이 중요하다.
 
내가 만들려는 Dialog의 유형은 다음과 같다.
아래처럼 가운데 영역만 자유롭게 구성할 수 있도록 설계해 볼 예정이다.
 
Dialog의 구조를 명확하게 하고 Dialog의 사용을 더욱 편리하고 일관되게 만들기 위해, 인터페이스를 정의해보자.
type AlertDialogButtonType = {
  label: string;
  action: () => void;
};

export interface DialogProps {
  isOpen: boolean;
  title: string;
  contents?: React.ReactNode;
  onConfirm: AlertDialogButtonType;
  onClose?: AlertDialogButtonType;
}
contents 프로퍼티는 Dialog 내부에 표시될 콘텐츠를 자유롭게 정의할 수 있게 ReactNode로 타입을 지정해주었다. 이로서 텍스트, 이미지, 또는 다른 React 컴포넌트를 포함할 수 있어 Dialog를 좀 더 유연하게 사용할 수 있게 된다.
 
버튼 타입도 따로 정의해두어, 각 버튼의 label과 action을 한 프로퍼티로 받을 수 있게 설계했다.
 

Dialog 컴포넌트 만들기

다음은 위의 interface를 토대로 구현한 Dialog 컴포넌트 코드이다.
// src/components/AlertDialog/AlertDialog.tsx
const AlertDialog = ({ isOpen, title, contents, onConfirm, onClose }: DialogProps) => {
  if (!isOpen) {
    return null;
  }

  return (
    <>
      <Overlay />
      <Container>
        <Text>{title}</Text>
        {contents}

        <ButtonGroup>
         {onClose && (
            <Button onClick={onClose.action}>{onClose.label}</Button>
          )}
          {onConfirm && (
            <Button onClick={onConfirm.action}>{onConfirm.label}</Button>
          )}
        </ButtonGroup>
      </Container>
    </>
  );
};
직관적으로 확인할 수 있도록 스타일링 코드는 생략하였고, isOpen이 true일 때만 DOM에 렌더링되도록 제작했다.
 

useState와 함께 사용해보기

이제 useState를 사용하여 Dialog의 등장을 제어해보자. 이 접근 방식은 간단하고 직관적이며 친숙한 패턴일 것이다.
import { useState } from 'react';
import AlertDialog from 'components/AlertDialog';

const HelloPage = () => {
  const [isOpen, setIsOpen] = useState<boolean>(false);

  return (
    <div>
      <button onClick={() => setIsOpen(true)}>Dialog 등장</button>

      <AlertDialog
        isOpen={isOpen}
        title="안녕하세요."
        onConfirm={{
          label: '확인',
          action: () => {
            setIsOpen(false);
          },
        }}
      />
    </div>
  );
};

export default HelloPage;
그러나 해당 패턴에서는 다양한 문제점이 존재할 수 있다.
 
만약, 한 페이지에 Dialog가 많이 필요하다면 useState로 모달 상태를 선언하는 부분이 많이 필요할 것이다.
또한 <Dialog /> 컴포넌트가 많아질 수록 return 부분의 UI 코드 길이가 비대해지는 문제가 발생할 것이다.
Dialog에 대한 흐름 제어 코드도 여기저기에 분산되어 있어 파악하기 어렵다.
 

개선해보기1: useDialog hook 제작

코드의 중복을 줄이고, 가독성을 높이기 위해, 반복되는 Dialog 관련 로직을 단순화할 필요가 있다.
 
특히 많은 Dialog 컴포넌트를 사용하는 페이지에서는 useState를 이용한 상태 관리 코드가 중복될 수 있다. 이를 해결하기 위해 useDialog라는 커스텀 훅을 만들어 보자.
// src/hooks/useDialog.tsx
const useDialog = (): [
  React.Dispatch<React.SetStateAction<boolean>>,
  (props: Omit<DialogProps, 'isOpen'>) => JSX.Element
] => {
  const [isOpen, setIsOpen] = useState<boolean>(false);

  const Dialog = (props: Omit<DialogProps, 'isOpen'>) => <AlertDialog isOpen={isOpen} {...props} />;

  return [setIsOpen, Dialog];
};

export default useDialog;
 
// src/pages/SendPage.tsx
function SendPage() {
	const [setIsOpen, Dialog] = useDialog();

	const handleSendMeessage = async () => {
		setIsOpen(true); // 👀
	};

	return (
		<>
			/** ... */
			<Button onClick={handleSendMeessage}>  // 👀
				보내기
			</Button>

			<Dialog
				title="확인해주세요"
				contents={<Text>{`00에게 보낼까요?`}</Text>}
				onConfirm={{ label: '보낼게요', action: () => { setIsOpen(false); trySendMessage(); } }} // 👀 
				onClose={{ label: '닫기', action: () => setIsOpen(false) }}  // 👀
			/>
		</>
	)
}
이처럼 useDialog 훅의 도입으로, Dialog의 상태 관리 및 호출 로직을 좀 더 간결하게 만들 수 있다.
 
그러나 이 방식은 여전히 UI 코드의 길이가 길어지는 문제와 Dialog 흐름 제어 코드가 분산될 수 있다는 문제를 해결하지는 못한다. 여전히 👀으로 주석처리한 부분들을 다 확인해야 Dialog의 흐름을 파악할 수 있다.
 

개선해보기2: useOverlay

이를 해결하기 위해, openDialog({ title: '', onConfirm: () => {} })와 같은 형태로 Dialog를 더 선언적으로 사용할 수 있도록 개선해보도록하자.

Overlay?

오버레이란 사용자 인터페이스에서 기본 UI 위에 추가적인 정보나 인터랙션을 제공하는 별도의 UI 레이어다. 이는 다양한 형태로 나타날 수 있으며, 대표적으로 BottomSheet, Dialog, Tooltip 등이 있다.
오버레이는 사용자 경험을 향상시키는 중요한 역할을 하며, 사용자의 주의를 필요한 정보나 작업에 집중시키기 위해 사용된다.

OverlayProvider 제작

OverlayProvider라는 Context Provider를 제작하여 오버레이 컴포넌트를 효과적으로 관리해보자.
// src/components/AlertDialog/OverlayProvider.tsx
interface OverlayContextProps {
  open: (overlayElement: React.ReactNode) => void;
  close: () => void;
}

export const OverlayDispatchContext = createContext<OverlayContextProps | null>(null);

const OverlayContext = ({ children }: PropsWithChildren) => {
  const [overlayComponent, setOverlayComponent] = useState<React.ReactNode>(null);

  const open = (overlayElement: React.ReactNode) => {
    setOverlayComponent(overlayElement);
  };

  const close = () => {
    setOverlayComponent(null);
  };

  const dispatch = {
    open,
    close,
  };

  return (
    <OverlayDispatchContext.Provider value={dispatch}>
      {children}
      {overlayComponent}
    </OverlayDispatchContext.Provider>
  );
};

export default OverlayContext;
// src/App.tsx
export default function App() {
  return (
		<OverlayContext>
			// ...
		</OverlayContext>
  );
}
이 Provider를 사용하면 애플리케이션의 어느 부분에서든 오버레이를 쉽게 추가하거나 제거할 수 있다.
open 함수를 호출하면 지정된 오버레이 컴포넌트가 화면에 렌더링되고, close 함수를 호출하면 오버레이가 제거된다.

useOverlay 훅 제작

useOverlay 훅을 통해 오버레이 컴포넌트를 열고 닫는 기능을 쉽게 사용해보자.
const useOverlay = () => {
  const overlayDispatch = useContext(OverlayDispatchContext);

  const open = (overlayElement: React.ReactNode) => {
    if (!overlayDispatch) {
      throw new Error('useOverlay must be used within an OverlayProvider.');
    }
    overlayDispatch.open(overlayElement);
  };

  const close = () => {
    if (!overlayDispatch) {
      throw new Error('useOverlay must be used within an OverlayProvider.');
    }
    overlayDispatch.close();
  };

  return {
    open,
    close,
  };
};

export default useOverlay;

사용해보자

‘사용자가 메시지 전송을 요청하면 AlertDialog는 onConfirm 액션을 통해 메시지 전송 함수를 호출하고 오버레이를 닫는다.’ 라는 요구사항을 구현해보자.
function SendMessagePage() {
  const overlay = useOverlay();
	
	const handleSendMessage = () => {
	    overlay.open(
	      <AlertDialog
	        isOpen
	        title="확인해주세요"
	        contents={<Text>{`00님께 메시지를 보낼까요?`}</Text>}
	        onConfirm={{
	          label: '보낼게요',
	          action: async () => {
	            overlay.close();
	            await sendMessage();
	          },
	        }}
	        onClose={{ label: '닫기', action: overlay.close }}
	      />
		);
	};

	return (
		//...
	);
}
이제 더이상 Dialog 제어 흐름이 여기저기 흩어져 있지 않으며 선언적으로 파악할 수 있게 되었다.

아쉬운 점

function SendMessagePage() {
  const overlay = useOverlay();
	
	const handleSendMessage = () => {
	    overlay.open(
	      <AlertDialog
	        isOpen
	        title="확인해주세요"
	        contents={<Text>{`00님께 메시지를 보낼까요?`}</Text>}
	        onConfirm={{
	          label: '보낼게요',
	          action: trySendMessage,
	        }}
	        onClose={{ label: '닫기', action: overlay.close }}
	      />
		);
	};

	const trySendMessage = async () => {
	    overlay.close(); // 떨어뜨려놓는다면..? 👿
	
	    await sendMessage();
	  };

	return (
		//...
	);
}
그러나 만약 overlay.close를 사용자가 떨어뜨려 놓는다면 Dialog 제어흐름을 파악하기 어려울 것이다.
이는 overlay.close가 onConfirm, onClose 작동 후에 실행될 수 있도록 수정하는 방향으로 개선해 볼 수 있을 것 같다.
 

개선해보기3: 포탈 사용

문제가 한 가지 더 있다. 바로 z-index 문제이다.
 
만약 Dialog의 z-index가 다른 UI 컴포넌트보다 낮을 경우, Dialog가 제일 최상단에 뜨지 않는다. 이를 매번 신경쓰기란 쉽지 않을 수 있다. Portal을 통해 개선해보도록 하자.
 
Portal은 리액트 프로젝트에서 컴포넌트를 렌더링하게 될 때, UI를 어디에 렌더링 시킬지 DOM을 사전에 선택하여 부모 컴포넌트의 바깥에 렌더링 할 수 있게 해주는 기능이다.
 
아까 작성했던 코드에 createPortal 로직을 추가해주기만 하면 된다.
import ReactDOM from 'react-dom';

const OverlayContext = ({ children }: PropsWithChildren) => {
  const [overlayComponent, setOverlayComponent] = useState<React.ReactNode>(null);

  const open = (overlayElement: React.ReactNode) => {
    // 오버레이 컴포넌트를 포탈을 사용해 렌더링합니다.
    const overlay = ReactDOM.createPortal(
      overlayElement,
      document.getElementById('dialog-root')! // 포탈 대상 엘리먼트
    );
    setOverlayComponent(overlay);
  };

  const close = () => {
    setOverlayComponent(null);
  };

  return (
    <OverlayDispatchContext.Provider value={{ open, close }}>
      {children}
      {overlayComponent} // 포탈을 통해 렌더링된 오버레이 컴포넌트
    </OverlayDispatchContext.Provider>
  );
};
 
 
코드를 보면 root가 아닌 dialog-root에 Dialog 컴포넌트가 들어가 있는 것을 볼 수 있다. 이로써 다른 UI 요소들과의 z-index 충돌을 방지할 수 있게 되었다.
 

추가 기능: 스크롤 동작 막기

오버레이가 나타난 상황에선 화면을 스크롤할 수 없도록 막아주는 기능이 필요한 경우도 있다.
 
이를 custom hook으로 구현해보자.
import { useEffect } from 'react';

/**
 * isOpen 상태에 따라 웹 페이지의 스크롤을 제어하는 역할을 합니다.
 * Dialog 열림 (isOpen이 true인 경우): body의 스크롤을 비활성화합니다.
 * Dialog 닫힘 (isOpen이 false인 경우): 기본 스크롤 동작을 복원합니다.
 */
const usePreventScroll = ({ isOpen }: { isOpen: boolean }) => {
  useEffect(() => {
    const bodyStyle = document.body.style;

    if (isOpen) {
      bodyStyle.overflow = 'hidden';
    } else {
      bodyStyle.overflow = '';
    }

    return () => {
      bodyStyle.overflow = '';
    };
  }, [isOpen]);
};

export default usePreventScroll;
// src/components/AlertDialog.tsx
const AlertDialog = ({ isOpen, title, contents, onConfirm, onClose }: DialogProps) => {
  usePreventScroll({ isOpen }); // 추가

  return (
    <>
			/** */
    </>
  );
};

export default AlertDialog;
 

Dialog가 열려있을 때

document.body.style.overflow 속성이 'hidden'으로 설정되어 body의 스크롤이 비활성화된다.
즉, Dialog가 열린 상태에서 사용자가 페이지를 스크롤하지 못하게 하여, 사용자가 Dialog의 내용에 집중할 수 있도록 할 수 있다.
 

Dialog 닫혀있는 경우

document.body.style.overflow 속성을 빈 문자열('')로 설정하여 기본 스크롤 동작을 복원한다.
따라서 Dialog가 닫힌 후 사용자가 페이지를 정상적으로 스크롤할 수 있게 된다.
 

마치며

요구사항에 따라 팀 스타일에 따라 추상화 레벨은 언제나 달라질 수 있으며, 코드에 정답은 없지 않은가. 아직 아쉬움이 남으며, 다음에 더 복잡한 Dialog를 마주한다면 더 개선시킬 수 있도록 구현해 봐야겠다.
